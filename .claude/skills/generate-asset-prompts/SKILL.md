# 에셋 원화 프롬프트 생성

프로젝트의 실제 아트 방향·기존 레퍼런스를 근거로, Nano Banana(주)와 ComfyUI(요청 시)에 바로 복사-붙여넣기 가능한 프롬프트 파일 모음을 만든다.

**왜 이렇게 하는가:** 매번 처음부터 프롬프트를 쓰면 캐릭터·세계관 디자인이 요청마다 미묘하게 달라진다. 레퍼런스 이미지로 형태를 고정하고, **색감·테마·무드**는 매 프롬프트에 반드시 적어 같은 세계 톤을 유지한다. 시간대·광원 같은 **조명 연출**만 별도 문단으로 빼도 된다.

## 하지 않는 것

- **레퍼런스 이미지를 실제로 안 보고(텍스트 문서만 읽고) 프롬프트를 쓰지 않는다.** art-direction.md 같은 문서는 "볼드한 외곽선"처럼 적혀 있어도, 실제로 승인되어 쓰이고 있는 레퍼런스 이미지는 그와 다를 수 있다(예: 아웃라인 없는 소프트 클레이 렌더). 이미지 자체가 문서보다 신뢰할 근거다 — 반드시 `Read`로 직접 열어본다.
- **문서 간 충돌이나 미결 사항을 발견해도 조용히 하나를 골라 넘어가지 않는다.** 프롬프트 파일에 `⚠️ 미결`로 남기고 채팅에서도 짚는다.
- **`Docs/Tasks/*.md`를 자동으로 새로 만들거나 수정하지 않는다.** PD가 별도로 요청하면 그때 한다.
- **다짜고짜 다 만들지 않는다.** 범위 항목이 비어 있으면 **반드시 `AskUserQuestion`** 으로 PD에게 묻는다. 채팅만으로 “이렇게 할게요”·“괜찮으세요?”·묵시적 기본값(키아트 추정, Nano 단독 확정 등)으로 넘어가지 않는다.

## 워크플로

### 0. PD 입력 게이트 (`AskUserQuestion` 필수)

사용자 메시지에서 아래 **5항목**을 파싱한다. **비어 있거나 애매한 항목마다** `AskUserQuestion`을 **한 번씩** 호출한다. 텍스트 질문만 보내는 것은 **금지**. 이전 항목이 확정될 때까지 **프롬프트 파일 Write 금지**.

| 순서 | 항목 | 사용자 메시지에 있으면 | 없으면 `AskUserQuestion` |
|------|------|------------------------|---------------------------|
| 1 | **에셋 종류** | 그대로 사용 | 옵션: 키아트/마케팅 장면, 캐릭터(표정·포즈), 배경·프롭, FX, UI 아이콘, 기타(Other로 직접 입력). **키아트를 Recommended로 두지 않는다.** |
| 2 | **개수** | 숫자 그대로 | 근거 수집(§2) **전**이면 art-direction·요청 맥락만으로 후보 2~3개 숫자 제시; **후**면 Read한 레퍼런스까지 반영해 Recommended 1개 + 대안. `allow_multiple: false`. |
| 3 | **플랫폼** | 명시대로 | Nano Banana만 / ComfyUI만 / 둘 다. **Recommended: Nano Banana만**(프로젝트 기본) — 단, **묻지 않고 확정하지 않는다.** |
| 4 | **색감·테마·무드** | PD 문구 그대로 프롬프트에 반영 | art-direction 키워드에서 뽑은 3~4옵션(예: 몽글 숲·파스텔 아이보리·코지·활기) + Other. **파일 작성 전 반드시 확정.** |
| 5 | **조명 연출** | 시간대·광원 지정 시 별도 문단 포함 | 별도 「조명 연출」 문단 **포함 / 생략(색감·무드만 본문에)** / **시간대 시리즈(낮·노을·밤 등)** 중 선택. |

**규칙**

- 한 턴에 여러 항목이 비어 있어도, **항목별로 `AskUserQuestion`을 분리**한다(한 폼에 다 넣지 않음). 단, 사용자가 이미 한 메시지에 5항목을 모두 명시했으면 게이트 생략하고 §2로 진행.
- §2 근거 수집 **후** 개수를 다시 제안해야 하면, 이미 확정된 개수와 충돌할 때만 **추가** `AskUserQuestion`(유지 vs 변경).
- 게이트가 끝나면 채팅에 **확정 범위 5줄 요약**을 보여준 뒤 §2로 간다. (요약만으로 재확인 `AskUserQuestion`은 필수 아님 — 저장 직전 §5에서 한 번 더 묻는다.)

### 1. 범위 파악 (확정값 기록)

§0에서 확정된 값만 표에 채운다. 임의 추정·기본값 덮어쓰기 금지.

| 항목 | 규칙 |
|------|------|
| **에셋 종류** | §0 확정값. 키 아트로 임의 추정하지 않는다. |
| **개수** | §0 확정값. |
| **플랫폼** | §0 확정값. |
| **조명 연출** | §0 확정값 — 별도 문단 포함 여부·시간대 시리즈 여부. |
| **색감·테마·무드** | §0 확정값 — 최종 영문 `프롬프트:`에 **반드시** 반영. art-direction·레퍼런스 Read·팔레트 HEX는 **이 톤을 구체화**하는 데 쓴다. PD가 §0에서 고르지 않았으면 **추론으로 채우지 말고** §0 질문부터 한다. |

### 2. 근거 수집 (아래 순서대로, 빠짐없이)

1. `planning/direction/art-direction.md` — 비주얼 키워드, 규칙 및 규격, 컨셉 아트 섹션
2. `planning/direction/design-pillars.md` — 대표 순간·서프라이즈·호기심 훅 (장면성이 있는 컨셉일수록 중요)
3. **레퍼런스 이미지를 `Read`로 실제로 연다** — `assets/아트 기조/`, `원화_테스트/`에서 다루려는 에셋과 가장 가까운 기존 결과물 1~3장. 텍스트 설명과 이미지가 다르면 이미지를 우선한다.
4. `Tasks/아트 컬러 팔레트 만들기.md` — 확정/후보 HEX가 있으면 **색감·테마·무드** 서술에 반영(별도 조명 문단 여부와 무관)
5. 관련 `Tasks/*.md`가 있으면 방향 메모를 참고만 한다(자동 수정 금지)

### 3. 프롬프트 골격 (Nano Banana)

매 컨셉은 아래 표 순서를 지킨다. **색감·테마·무드는 생성 요청에 대한 답으로 파일을 쓸 때 항상** `프롬프트:` 본문 어딘가에 들어가야 한다(전용 문단이 아니어도 됨). 자세한 스켈레톤은 [template.md](template.md).

| 단계 | 필수 | 내용 |
|------|------|------|
| **에셋 고정** | ✓ | "Use the [character/asset] shown in the reference image exactly as-is" + 이미지에서 관찰한 재질·색·선 유무. **색감·무드**는 여기 **소재 색**만이 아니라, 아래 항목과 겹치지 않게 최소 한 줄은 다른 문단에도 반드시 있다. |
| **장면/구도** | ✓ | 장면·카메라([[Tasks/카메라 렌더링 라이팅 기술 방향]]), 주변 요소. 무드가 장면과 묶이면 **색감·테마·무드**를 여기에 넣어도 된다. |
| **조명 연출** | 선택 | **광원·시간대·그림자·Bloom 느낌** 등 라이팅 셋업만. PD가 시간대를 지정했거나 키아트 **무드 시리즈**(낮/노을/밤)일 때 별도 문단 권장. 단일 프롭·UI·턴어라운드는 **이 문단만 생략** 가능 — 단, **색감·테마·무드 생략은 불가**. |
| **스타일 앵커** | ✓ | 레퍼런스와의 일관성 + **몽글 숲 마을·하프 판타지·파스텔/아이보리 톤** 등 프로젝트 테마가 빠지지 않게. 조명 문단을 생략했으면 무드·색감은 여기 또는 장면에 명시. |
| **기술 조건** | ✓ | 비율, 텍스트/워터마크 금지, 등장 개체 수 등 |

**자가검수(파일 저장 전):** `프롬프트:` 전체를 읽고 색감(color palette / saturation / warmth), 테마(world fantasy tone), 무드(cozy / playful / dreamy 등)가 **영문으로 최소 1문장 이상** 보이는지 확인한다.

### 4. ComfyUI 변환 (요청 시에만)

Nano Banana 프롬프트를 그대로 재사용하지 않는다 — ComfyUI의 CLIPTextEncode는 서술형 문장보다 **쉼표로 구분된 구/키워드** 조합에 더 안정적으로 반응한다.

- **Positive**: Nano 골격을 쉼표 구로 압축. **색감·테마·무드 키워드는 항상 포함**(예: `pastel ivory`, `cozy forest village`, `cheerful saturation`). 조명 연출은 별도 문단을 썼을 때만 `golden hour`, `soft dappled light` 등 추가.
  예: `..., mossy forest floor, pastel green and cream palette, cozy half-fantasy mood, soft 3D clay-render style`
- **Negative**: 스타일에서 명시적으로 배제해야 할 것을 근거 문서·이미지에서 역산해 채운다.
  예: 레퍼런스가 "아웃라인 없음"으로 확정돼 있다면 `outline, line art, cel shading lines, sketch lines` 를 negative에 넣어 모델이 임의로 선을 추가하지 않게 한다.
- ComfyUI는 Nano Banana와 달리 이미지 레퍼런스를 프롬프트에 직접 못 넣는다(IPAdapter 등 별도 노드 필요) — 이 사실을 README에 한 줄로 안내만 하고, 노드 구성 자체는 이 스킬 범위 밖임을 밝힌다.

### 5. 출력

**저장 직전:** `00-README`·`NN-*.txt`를 Write하기 **바로 전** `AskUserQuestion` 1회 — 폴더명(`assets/{에셋명} 프롬프트/`), 확정 개수, 플랫폼, ComfyUI 파일 유무, 우선 컨셉 후보 1개를 옵션에 넣고 **(Recommended) 이대로 저장** / 수정 요청 / 취소.

- **폴더**: `assets/{에셋명} 프롬프트/` (한글, 기존 `assets/키아트 프롬프트/`와 같은 패턴)
- **`00-README.txt`**: 사용법, 각 파일이 참조하는 이미지 경로, 결과 저장 위치 안내(`원화_테스트/`에 컨셉 번호를 포함해 저장)
- **`NN-{컨셉명}.txt`**: 컨셉별 파일. [template.md](template.md)의 스켈레톤을 따른다 — `[컨셉]` / `근거` / `참조 이미지` / `프롬프트:` 블록, 미결 사항은 `⚠️ 미결` 줄 추가.
- 여러 버전(예: 시간대 3종, 감정 3종)이 필요한 컨셉은 한 파일 안에 "공통 베이스 + 버전별 교체 문장" 구조로 묶는다 — 파일 수가 불필요하게 늘어나지 않게.

### 6. 마무리

- 채팅에 만든 파일 목록과, 그중 프로젝트 훅(design-pillars 대표 순간 등)과 가장 잘 맞는 컨셉 1개를 우선순위로 짚어준다.
- 근거 수집 중 발견한 미결/충돌 사항은 파일뿐 아니라 채팅에서도 별도로 언급한다.
- TaskNotes 연동은 하지 않는다 — PD가 필요하면 별도로 요청한다.

## 예시

**PD**: "에셋 원화 프롬프트 좀 만들어줘"
→ §0: 에셋 종류 → 개수 → 플랫폼 → 색감·테마·무드 → 조명 연출 순으로 **각각 `AskUserQuestion`**. 키 아트로 추정하지 않음. 전부 확정·§5 저장 직전 확인 후에만 파일 생성.

**PD**: "몬스터 하우스 배경 프롭 원화 3개만 nano banana로 만들어줘"
→ 종류·개수·플랫폼은 메시지에 있음. **색감·테마·무드**·**조명 연출**만 §0에서 `AskUserQuestion` → §2~§5.

**PD**: "캐릭터 표정 배리에이션 5개, comfyui용으로도"
→ 종류·개수·플랫폼 확정. **색감·테마·무드**·**조명 연출** §0 질문 후 ComfyUI 파일 포함해 저장(§5 확인).
