# Generate Spec — 출력 템플릿 & 형식 규칙

`SKILL.md` 절차에서 호출하는 출력 형식 정본. 새 스펙 파일은 이 형태로만 만든다.

## 표기 (범례)

| 표기 | 뜻 | 어디서 쓰는가 |
| --- | --- | --- |
| `+ 추가` | 새 블록 | 블록 헤더 |
| `/ 변경` | 기존 흐름을 손대는 블록 | 블록 헤더 |
| `- 제거` | 빼는 블록 | 블록 헤더 (필요할 때만) |
| `~ 현재` | 지금 코드/에셋에 이미 있는 사실 | 블록 안 줄 |
| `! 문제` | 그 사실이 기획서를 충족 못 시키는 이유 | 블록 안 줄 |
| `> 할 일` | 이번 스펙에서 해야 하는 일 | 블록 안 줄 |
| `x 금지` | 이 스펙에서 명시적으로 안 하는 것 | 블록 안 줄 / 하지 않는 것 |

## 줄 작성 규칙

- 한 줄 = 동작 1개. “A 하고 B 한다” 금지 — 줄을 쪼갠다.
- 클래스명/필드명/네임스페이스 노출 금지. 자연어로 환원.
- 외부 에셋 식별자(상태 이름, 프리팹 경로, 컨트롤러 파일명)는 인용 OK.
- 블록 이름은 시스템 클래스명이 아니라 **사용자가 보는 이름**(예: “스케줄 상태 연결”, “캐릭터 초기 상태”).
- 없는 것은 적지 않는다. `! 문제` 가 떠오르지 않으면 그 줄은 통째로 비운다.
- “시니어가 이 줄만 보고 다음에 뭘 만질지 감이 오는가?” — 안 오면 더 쪼갠다.

## 보조 섹션 결정

| 섹션 | 둘 때 |
| --- | --- |
| **행동 → 애니메이션 매핑** (1:1 표) | 외부 에셋 위에서 작업하면 거의 항상 |
| **/ 에셋 기준** | 외부 에셋이 스펙 핵심일 때 |
| **/ 하지 않는 것** | 항상 (범위 잠그기) |
| **/ 완료 기준** | 항상. 화면/로그에서 검증 가능한 사실로만. 클래스명·내부 상태명 금지. prefix 없이 `-` 리스트. |

## 출력 템플릿

```markdown
## v0 <주제> 스펙

### 기준 기획서

- `Docs/planning/<주제>.md`

### 목표

<한 문단. 이 스펙이 끝나면 화면에서 무엇이 보이는가.>

### 표기 (범례)

`+ 추가` `/ 변경` `- 제거` `~ 현재` `! 문제` `> 할 일` `x 금지`

---

## 작업 목록

/ 변경 **<블록 이름>**
- ~ <지금 있는 것>
- ! <왜 부족한지>
- > <해야 할 일 1개>
- > <해야 할 일 1개>
- x <이번 스펙에서 안 하는 것>

+ 추가 **<블록 이름>**
- ~ 없음.
- > <해야 할 일 1개>
- > <해야 할 일 1개>

## 행동 → 애니메이션 매핑   <!-- 외부 에셋 위에서 작업할 때만 -->

| 기획 행동 | 재생할 상태 |
| --- | --- |
| ... | `...` |

---

## / 에셋 기준   <!-- 해당될 때만 -->

- ~ `<에셋 이름>` — <사실 한 줄>
- > <이번 스펙에서 어떻게 쓰는지>
- x <건드리지 않는 부분>

---

## / 하지 않는 것

- x <범위 밖 1>
- x <범위 밖 2>

---

## / 완료 기준

- <화면/로그에서 확인 가능한 사실 1>
- <화면/로그에서 확인 가능한 사실 2>
```

## 자주 틀리는 형식

- 클래스/필드/네임스페이스를 본문에 적는다 → 자연어로 다시.
- 한 줄에 두 가지 동작을 넣는다 → 줄을 쪼갠다.
- 기획서에 없는 새 규칙을 스펙에서 만든다 → 멈추고 기획서로 보낸다.
- 같은 주제로 여러 스펙 파일을 만든다 → 항상 1주제 1파일. 이전 파일은 통합 후 삭제 후보.
- 코드 1패스를 깊이 읽기로 바꾼다 → 1패스는 “현재 한 줄 요약”용이지 설계용이 아니다.
- 외부 에셋 사실을 추정으로 적는다 → MCP/실제 파일에서 확인한 것만 적는다.
- `! 문제` 자리를 의무적으로 채운다 → 없으면 비운다.
- `/ 완료 기준` 에 prefix 를 붙인다 → 이 섹션만 prefix 없이 `-` 리스트.
