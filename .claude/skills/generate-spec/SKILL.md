# Generate Spec — 컨텐츠 1개 → 스펙 1개

기획서를 입력으로 받아 스펙만 작성한다.
기획서 작성과 수정은 다루지 않는다.

**출력 형식·줄 작성 규칙·템플릿은 [`template.md`](template.md) 정본.** SKILL.md 는 절차만 다룬다.

## 0. 적용 범위

- **입력:** 잠긴 **콘텐츠·체험** 기획서 1개 (`Docs/planning/<주제>.md`). 수치·시드·에셋 경로가 planning에 없어도 됨 — spec 작성 시 코드 1패스·기존 spec·PD 결정으로 채움. planning에 구현 힌트만 있으면 spec으로 **이관**하고 planning 갱신은 마나 위임.
- **출력:** 같은 주제의 스펙 1개 (`Docs/specs/<주제>.md`). 동일 주제로 여러 스펙 파일 만들지 않는다.
- **다루지 않는 것:** 코어루프·페이싱 문서, narrative 문서, 기획서 본문 수정, 코드/씬/테스트 작성.

## 1. 절차

### 1) 입력 잠그기

- `Docs/planning/<주제>.md` 1개를 정본으로 고른다. 여러 후보면 멈추고 어느 기획서가 정본인지 묻는다.
- 기존 `Docs/specs/<주제>.md` 가 있으면 읽어 어떤 부분을 갱신/대체할지 정한다.

### 2) 외부 에셋 인스펙션 (해당될 때만)

기획서가 외부 에셋(Unity 프리팹·AnimatorController, 외부 라이브러리 등) 위에 얹힌다고 명시하면, 코드를 보기 전에 **에셋부터** 본다.

- Unity 에셋: `user-unityMCP` 의 `manage_animation`, `manage_asset`, `manage_gameobject` 등을 사용해 레이어/상태/파라미터/프리팹 컴포넌트를 추출한다.
  - 사용 전 `mcps/user-unityMCP/tools/<tool>.json` 스키마를 먼저 읽는다.
  - 다중 인스턴스 환경이면 `set_active_instance` 로 대상 고정.
- 추출한 사실(상태 이름, 기본 상태, 자동 전이 유무, 파라미터 유무, 보유 컴포넌트 등)만 스펙에 적는다. 추정·해석은 적지 않는다.

### 3) 현재 코드 1패스

- 기획서가 가리키는 시스템 주변만 Glob/Grep 1패스로 훑는다. 자세히 읽지 않는다. **목적은 “지금 화면/로직에서 무엇이 이미 일어나고 있는가”** 한두 줄 요약.
- 클래스명·메서드 시그니처를 스펙에 옮기지 않는다. “시간에 따라 상태가 바뀌는 구조는 있다” 처럼 자연어로 환원.

### 4) 작업 단위로 분해

기획서 한 항목씩 보면서 **`~ 현재` / `! 문제` / `> 할 일` / `x 금지`** 4종 줄로만 환원한다.
줄 작성 세부 규칙은 [`template.md`](template.md) 의 “줄 작성 규칙” 정본.

### 5) 그룹핑

작업 줄을 **`/ 변경` / `+ 추가` / `- 제거`** 블록으로 모은다. 블록 이름은 사용자가 보는 이름으로 짓는다.

### 6) 보조 섹션

`행동 → 애니메이션 매핑`, `/ 에셋 기준`, `/ 하지 않는 것`, `/ 완료 기준` 중 필요한 것만 둔다.
어떤 섹션을 어떤 조건에서 두는지는 [`template.md`](template.md) 의 “보조 섹션 결정” 표 참고.

### 7) 정합 점검

스펙 파일을 닫기 전에:

- 기획서에 있는데 스펙에 없는 항목이 있는가? 있으면 추가하거나, 의도적으로 v0에서 빼는 거면 `x` 로 명시한다.
- 스펙에만 있고 기획서에 근거 없는 항목이 있는가? 있으면 지운다. 새 규칙은 기획서로 돌아가서 잡는다.
- 코드 클래스명이 본문에 남아있지 않은가? 남아있으면 자연어로 환원.
- `Docs/specs/` 안에 같은 주제 다른 파일이 남아있지 않은가? 있으면 통합/삭제 후보로 보고한다.

## 2. 출력 형식

[`template.md`](template.md) 그대로 따른다. 절차에서 만든 4종 줄 + 블록 + 보조 섹션을 그 템플릿에 박는다.

## 3. 금지

- `Assets/` 코드·씬·프리팹 수정.
- 기획서(`Docs/planning/`) 본문 수정. 갭이 보이면 game-designer(마나) 에게 위임.
- 같은 주제의 추가 스펙 파일 생성.
- 코드 클래스명·시그니처를 본문에 넣는 것.
- “하지 않는 것” 없이 범위를 열어두는 것.
