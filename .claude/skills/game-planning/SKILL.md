# 게임 기획서 작성

게임에서 플레이어가 겪는 내용만 기획한다.
개발 전달, 작업 관리, 코드 구조는 다루지 않는다.

## 작성 순서

1. 사용자가 정한 방향과 기존 게임의 모습을 확인한다.
2. 핵심 선택이 비어 있으면 작성 전에 질문한다.
3. 아래 네 장으로 기획서를 작성한다.
4. 완성 후 어려운 말, 반복, 다른 문서 언급을 지운다.

## 문서 구조

```markdown
# [기획 이름]

## 오버뷰
[무엇에 대한 기획인지 2~4문장]

## 의도
[왜 필요한지와 플레이어가 느껴야 할 것]

## 기획 본문
[플레이어가 보는 것, 하는 것, 게임의 반응]

## 필요한 기능
[기획을 성립시키는 기능의 이름과 플레이어에게 보이는 역할]
```

`기획 본문`은 내용에 맞는 쉬운 이름으로 바꿀 수 있다.

## 반드시 지킬 것

- 오버뷰, 의도, 기획 본문, 필요한 기능의 네 장만 쓴다.
- 짧고 쉬운 한국어를 쓴다.
- 한 문장에는 한 가지 내용만 쓴다.
- 실제 게임 장면과 행동을 중심으로 쓴다.
- 문서 하나만 읽어도 이해되게 쓴다.
- 기획을 성립시키는 기능을 짧게 정의한다.
- 기능은 이름과 플레이어에게 보이는 역할만 적는다.

## 쓰지 않을 것

- 다른 문서의 경로, 링크, 이름
- 전문용어, 영어 약어, 내부 코드명
- 방향을 대신하는 키워드 나열
- 출처와 참고 자료
- 담당 영역과 문서 사이의 관계
- 포함, 제외, 범위 목록
- 개발 방법, 코드, 데이터 구조
- 스펙, 작업, 일정, 검수, 전달 절차
- 이론 이름과 형식적인 분석 틀

기존 문서는 배경을 이해하기 위해서만 읽는다.
완성된 기획서가 다른 문서를 가리키거나 다른 문서를 알아야 이해되는 형태가 되면 안 된다.
