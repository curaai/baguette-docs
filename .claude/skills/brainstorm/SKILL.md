# 기획 아이디어 브레인스토밍

컨셉·컨텐츠 **기획서용 아이디어**를 고른다.  
전체 게임 비전·`game-concept`·필러 파이프라인은 다루지 않는다.  
킥오프 대체도 아니다 — `planning-kickoff`와 별개로, 막히거나 PD가 요청할 때만.

**파일로 export하지 않는다.** 산출은 채팅에 정리한 텍스트뿐이다.  
풀 기획서(`game-planning` 본문)도 쓰지 않는다. 결정은 **AskUserQuestion**.

## 순서

### 1. 기획 대상·위상부터 적극 묻기

주제가 한 줄로 와도 **바로 후보를 내지 않는다.**  
관련 foundation·기존 planning을 짧게 읽은 뒤, AskUserQuestion으로 아래를 **먼저** 잠근다(텍스트만 "뭐 할까요?" 금지).

- **이번에 기획할 것** — 시스템 / 컨텐츠 장면 / 아이템·오브젝트 / 적·조우 / 공간·동선 / 기타(직접 입력). 한 메시지에 여러 개가 섞여 있으면 무엇을 **이번**에 할지 고르게 한다.
- **위상** — 이 조각이 게임에서 어디에 서는지. 예: 코어 루프의 한 축 / 한 구역만의 양념 / 진행 게이트 / 분위기·연출 / 아직 모름. 옵션은 맥락에 맞게 만들고, 항상 자유 서술 탈출구를 둔다.
- **지금 막힌 지점**(있으면) — 방향이 없음 / 후보는 있는데 고르기 어려움 / 톤만 맞추고 싶음 / 기타.

답이 비거나 추측이 필요하면 추정으로 메우지 말고 다시 묻는다.  
잠긴 내용을 한 줄 브리프(대상 + 위상)로 읽어 준 뒤 2로 간다.

### 2. 방향 많이 내기

서로 다른 방향 **4~6개**를 낸다. 비슷한 변형을 쪼개지 말고, 축이 다르게 갈라지게 한다.  
각 후보: 이름 + 한두 문장 + 맛/리스크 한 줄 + **이 위상에 맞는지** 한 마디.

AskUserQuestion: 하나 고르기 / 여러 개 섞기 / 더 뽑기 / 대상·위상부터 다시.

### 3. 채팅에 정리

승인 후 **채팅 메시지 하나**로만 정리한다. Write·파일 생성 금지.

```markdown
# [주제] 아이디어

## 기획 대상
[이번에 기획할 것]

## 위상
[게임 안에서의 자리]

## 이번에 정할 것
[1~2문장]

## 고른 방향
[이름] — [요약]

## 다른 후보
- …

## 다음에
기획서 본문은 `game-planning` / 새 주제 시작은 `planning-kickoff`
```

끝. **COMPLETE**

## 하지 않을 것

- 대상·위상 확인 전에 후보부터 쏟아내기
- `*-ideas.md` 등 어떤 경로든 아이디어 파일 Write
- `Docs/planning/**` 풀 기획서 Write
- 킥오프 Phase를 대신 진행
- 전체 게임 컨셉·필러·비주얼 앵커 문서화
